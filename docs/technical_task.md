## 2. ОБЩИЕ СВЕДЕНИЯ

### 2.1. Наименование проекта
Информационная система для магазина обуви ООО «Обувь»

### 2.2. Цель проекта
Создание информационной системы для автоматизации учета товаров и заказов, а также разграничения прав доступа пользователей в соответствии с их ролями.

### 2.3. Задачи проекта
- Обеспечить возможность просмотра каталога товаров для всех категорий пользователей (включая гостей)
- Реализовать модуль авторизации с разграничением прав доступа
- Создать интерфейс управления товарами для администратора (добавление, редактирование, удаление)
- Реализовать модуль управления заказами для менеджера и администратора
- Обеспечить фильтрацию, сортировку и поиск товаров для менеджера и администратора

### 2.4. Планируемый результат
Веб-приложение, обеспечивающее удобный просмотр каталога обуви, управление товарами и заказами с учетом ролей пользователей.

## 3. ТРЕБОВАНИЯ К ФУНКЦИОНАЛУ СИСТЕМЫ

### 3.1. Основные функции системы
- Авторизация пользователей по логину и паролю
- Гостевой вход (просмотр товаров без авторизации)
- Просмотр товаров (для всех ролей)
- Фильтрация, сортировка и поиск товаров (для менеджера и администратора)
- Добавление/редактирование/удаление товаров (только для администратора)
- Просмотр заказов (для менеджера и администратора)
- Добавление/редактирование/удаление заказов (только для администратора)

### 3.2. Роли пользователей

**Таблица №2 - Роли и права доступа**

| Роль | Описание | Права доступа |
|------|----------|---------------|
| Гость | Неавторизованный пользователь | Просмотр товаров (без фильтрации, сортировки, поиска) |
| Авторизованный клиент | Зарегистрированный покупатель | Просмотр товаров (без фильтрации, сортировки, поиска) |
| Менеджер | Сотрудник магазина | Просмотр товаров (с фильтрацией, сортировкой, поиском), просмотр заказов |
| Администратор | Полный доступ к системе | Полный CRUD товаров и заказов, просмотр с фильтрацией/сортировкой/поиском |

### 3.3. Авторизация
- Вход по логину (email) и паролю
- Возможность продолжить как гость (без авторизации)
- После успешной авторизации открывается главное окно с соответствующими правами

## 4. НЕФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ

### 4.1. Требования к интерфейсу
- Все формы должны иметь заголовок
- На главной форме установлен логотип (без искажений, изменений пропорций и цвета)
- Шрифт: Times New Roman
- Основной фон: #FFFFFF (белый)
- Дополнительный фон: #7FFF00
- Цвет акцентирования внимания: #00FA9A
- Если размер скидки превышает 15%, фон товара: #2E8B57

### 4.2. Безопасность
- Пароли должны храниться в хешированном виде
- Защита от SQL-инъекций через параметризованные запросы
- Разграничение доступа на основе ролей

## 5. ТЕХНИЧЕСКИЕ ТРЕБОВАНИЯ

### 5.1. Технологический стек
- Backend: Python (Flask/Django)
- Frontend: HTML, CSS
- База данных: Supabase (PostgreSQL)
- Хостинг: GitHub Pages / Vercel / любой хостинг с поддержкой Python

## 6. ТРЕБОВАНИЯ К ДАННЫМ

### 6.1. Сущности БД
- **Товары** (артикул, наименование, цена, категория, производитель, поставщик, скидка, количество на складе, описание, фото)
- **Пользователи** (ФИО, логин/email, пароль, роль)
- **Заказы** (номер заказа, дата заказа, дата доставки, адрес пункта выдачи, ФИО клиента, код получения, статус)
- **Состав заказа** (связь заказа с товарами и их количеством)
- **Пункты выдачи** (адрес)

## 7. КРИТЕРИИ ПРИЁМКИ

- Реализован вход для всех ролей согласно ТЗ
- Гость видит товары без возможности фильтрации
- Менеджер видит товары с фильтрацией и заказы
- Администратор может добавлять/редактировать/удалять товары и заказы
- Соблюдены требования к цветовой гамме и шрифтам
- Скидка >15% выделяется цветом #2E8B57
